# Streamlit 移动端响应式布局设计

## 1. 目标

本次改造为现有 Streamlit 评测项目增加手机和平板适配，使用户能够在移动端完整浏览样本、模型回答、AI 评分和评测结论，并能够选择样本与模型。真实评测、继续评分、失败重试等操作继续保留，但在移动端降低视觉权重并加强状态提示和确认边界。

改造只调整响应式布局、控件排列和内容溢出处理，不修改：

- 任何现有项目文案；
- 数据集题目、标准答案和样本内容；
- 评分模型、裁判 Prompt、评分标准或评分逻辑；
- 模型调用、失败恢复和 Token 使用逻辑；
- PostgreSQL/Supabase 表结构、增量写入和持久化读取逻辑；
- 当前已持久化的模型回答和评分结果。

## 2. 已确认的功能分层

移动端采用“阅读优先、关键操作保留”的分层方案：

1. 完整支持浏览项目说明、样本、模型回答、AI 评分和评测结论。
2. 完整支持选择样本和选择模型。
3. 保留运行评测、继续评分、失败重试和放弃运行等操作。
4. 运行评测作为发起评测页的主操作；重试、放弃等次要或高风险操作保留在对应内容区，不放入全局固定操作栏。
5. 不为移动端建立独立页面、独立路由或第二套状态，桌面端和移动端共用相同组件、会话状态和持久化数据源。

## 3. 方案选择

采用“横向滑动导航 + 单列内容 + 评测页底部主操作”的方案。

### 3.1 选择理由

- 与当前 Streamlit 页面结构兼容，改动范围可控；
- 导航占用高度小，适合长回答和长结论；
- 单列内容可以保持稳定的阅读顺序；
- 底部主操作便于单手触达，同时只出现在需要运行评测的页面；
- 不需要复制导航或维护两套页面状态。

### 3.2 未采用方案

- 不采用 2×2 导航入口：入口清晰，但长期占用首屏空间，会把回答和评分内容继续向下推。
- 不采用固定底部四项导航：单手操作更接近原生 App，但在 Streamlit 中容易与软键盘、弹窗、浏览器安全区域和评测主操作竞争空间，兼容风险较高。

## 4. 响应式断点与总体结构

响应式断点沿用并收敛当前项目已有的 `860px` 和 `760px` 规则，避免新增相互冲突的断点。

### 4.1 桌面端：大于 860px

- 保持当前页面结构和视觉密度；
- 保持现有多列配置、工具栏和数据表行为；
- 不因移动端改造改变桌面端的文案、控件顺序或业务交互。

### 4.2 平板端：761px–860px

- 缩小页面水平留白和列间距；
- 双列内容允许自然换行；
- 导航保持完整，但在空间不足时可横向滑动；
- 数据表使用自身滚动容器，不扩大页面整体宽度。

### 4.3 手机端：不大于 760px

- 所有主要内容改为单列；
- 页面标题正常随内容滚动；
- 顶部页面导航变为单行横向滑动标签栏，并在页面滚动时吸附于顶部；
- 当前页面标签必须处于可见区域，并保留明确的选中状态；
- 表单控件、选择器和主要按钮占满可用宽度；
- 页面水平留白缩小，但正文保留足够的阅读边距。

### 4.4 小屏手机：不大于 480px

- 进一步收紧标题字号、卡片内边距和区块间距；
- 多按钮操作组改为纵向排列；
- 小型状态指标允许以紧凑的两列网格展示，但不改变阅读顺序；
- 可点击控件保持至少约 44px 的触控高度。

## 5. 导航与页面框架

当前固定比例的五列导航在手机端改为：

1. 项目名称作为页面品牌行保留；
2. 四个页面入口保持原有文案和顺序；
3. 页面入口位于一个可横向滑动的标签栏中；
4. 标签栏吸附在 Streamlit 顶部工具区下方，不遮挡系统工具栏；
5. 标签栏仅横向滚动，不触发整个页面横向滚动；
6. 当前页面标签使用现有主题强调色，并在页面切换后自动回到可见范围。

导航仍调用现有页面切换逻辑，不新增路由状态。

## 6. 各页面手机端设计

### 6.1 项目说明

- 内容按当前顺序单列展示；
- 说明卡片、流程区块和提示信息占满可用宽度；
- 不压缩或重排现有文案；
- 图标和装饰元素可以缩小，但不得隐藏实际说明内容。

### 6.2 样本库

- 页面标题和筛选条件优先展示；
- 新增样本、导入 CSV、同步样本库等低频操作排列在筛选区之后；
- 普通手机上的操作按钮纵向排列，小屏不会出现按钮文字挤压；
- 样本数据表保留现有字段和行数据，表格自身允许横向滑动；
- 关键识别字段保持在表格前部，不通过缩小字号强塞全部列；
- 样本详情改为单列，长文本正常换行；
- 编辑、删除和同步等操作不使用固定底栏；
- 导入、新增、编辑和确认移出测试弹窗在手机端接近全屏，双列表单改为单列。

### 6.3 发起评测

- “评测配置”“模型回答”“AI 评分”保持当前顺序并单列展示；
- 选择样本和选择模型按钮在手机端纵向排列；
- 配置摘要使用紧凑卡片，不隐藏预计回答数量、模型数量或任务数量；
- “查看持久化批次”选择器位于模型回答区域顶部；
- 模型回答采用现有“选择器 + 回答详情”结构，一次聚焦一个回答，避免把大量长文本同时铺开；
- 回答全文、技术明细和评分对比弹窗在手机端接近全屏；
- 评分表和技术明细表使用独立横向滚动区域；
- 失败状态、未完成状态、可继续状态和持久化状态完整显示。

### 6.4 运行评测操作区

- 仅发起评测页的“运行评测”主操作进入底部吸附操作区；
- 操作区保留现有按钮文案、禁用条件、运行状态和点击逻辑；
- 操作区使用 `env(safe-area-inset-bottom)` 兼容 iPhone 安全区域；
- 页面内容增加与操作区等高的底部留白，确保最后一段回答或评分不被遮挡；
- 输入控件获得焦点或弹窗打开时，吸附操作区不得覆盖软键盘或弹窗操作；
- 继续评分、重试失败项、放弃本次运行等操作留在对应状态卡片内；
- 放弃等高风险操作继续使用现有确认流程，不因移动布局减少确认步骤。

### 6.5 评测结论

- 数据来源提示、当前结论、模型问题详情按当前顺序单列展示；
- 结论摘要和模型详情占满可用宽度；
- 模型比较表保留现有数据，但在自身容器内横向滑动；
- 用户先选择模型，再查看对应问题、模型回答和评分详情；
- 模型回答正文保持完整，不因手机端适配截断数据库内容；
- 数据维护入口继续保留，但不作为移动端首要操作。

## 7. 长内容、数据表与弹窗

### 7.1 长回答

- 保留现有 Markdown 渲染和原始内容；
- 普通文本使用自然换行；
- 长 URL、连续字符和行内代码允许断行或局部滚动；
- 代码块使用自身横向滚动区域；
- 不用固定高度截断回答，也不修改数据库中的回答文本。

### 7.2 数据表

- 页面容器保持 `max-width: 100%`；
- Streamlit dataframe 容器独立横向滚动；
- 不通过极小字号展示全部列；
- 不在移动端删除样本、回答、评分、错误或持久化字段；
- 表格滚动不影响顶部导航和页面纵向滚动。

### 7.3 弹窗

- 手机端宽度接近视口宽度并保留约 12px 外边距；
- 内容区允许纵向滚动；
- 原双列表单改为单列；
- 确认与取消按钮保持可见、可点击且顺序一致；
- 弹窗关闭后保留现有页面状态和已选择内容。

## 8. 实现边界

响应式实现集中在现有 UI 层：

- `src/ui/components.py`：全局响应式样式、通用内容溢出和移动端操作区样式；
- `src/ui/navigation.py`：移动端横滑导航结构和稳定的组件标识；
- `src/ui/samples.py`：样本工具栏、表格、详情和弹窗的响应式排列；
- `src/ui/test_run.py`：配置区、样本与模型选择、回答、评分及底部主操作区；
- `src/ui/conclusions.py`：结论摘要、模型详情和数据表的响应式排列。

实现应优先使用组件专属 key 或稳定的语义容器标识，避免使用依赖 Streamlit 内部节点顺序的宽泛 `nth-child` 选择器。不得为移动端重复渲染整套页面，也不得建立独立的移动端会话状态。

## 9. 数据流与错误处理

本次改造不改变数据流：

1. 页面继续从当前会话状态和 PostgreSQL/Supabase 读取持久化批次；
2. 选择批次后继续按现有逻辑读取模型回答和 AI 评分；
3. 运行评测、增量写入、失败停止和恢复机制保持不变；
4. 响应式样式加载失败时，页面仍应以 Streamlit 默认布局提供全部功能；
5. 移动端不得把数据库错误、模型失败或未完成状态隐藏在折叠区之外；
6. 不在本次布局验证中触发新的模型回答或 AI 评分。

## 10. 测试与验收

### 10.1 自动化回归

- 运行现有全量测试、Ruff 和项目数据校验；
- 增加响应式结构测试，验证导航、关键按钮、持久化批次选择器、模型回答和评分区域仍存在；
- 验证样本选择、模型选择和运行评测的现有状态逻辑没有改变；
- 验证持久化读取和结论计算测试保持通过。

### 10.2 视觉与交互检查

至少使用以下视口检查四个页面及主要弹窗：

- `390×844`：常见 iPhone 宽度；
- `430×932`：大屏手机；
- `768×1024`：平板；
- `1440×900`：桌面回归。

每个视口检查：

1. 页面不存在意外的整体横向滚动；
2. 导航标签可访问且当前页面状态清楚；
3. 页面标题、样本、回答、评分、错误状态和持久化信息没有被遮挡；
4. 底部运行评测操作区不覆盖最后一段内容；
5. 数据表和代码块只在自身区域横向滑动；
6. 主要按钮和选择器可以正常触控；
7. 长回答、长链接和失败信息不会撑破卡片；
8. 桌面端布局没有响应式回归。

### 10.3 线上验收

1. 本地完成自动化与四档视口检查；
2. 将实现提交并合并到 `main`；
3. 推送 GitHub 并等待 CI 成功；
4. 检查 Streamlit Community Cloud 完成最新提交部署；
5. 在线上应用复核桌面和手机视口；
6. 核对线上仍可读取当前持久化回答和评分；
7. 不以新的模型调用作为布局验收手段。

## 11. 明确不做

- 不修改任何现有项目文案；
- 不修改评分模型、评分口径或结论规则；
- 不重新生成模型回答或 AI 评分；
- 不删除、迁移、复制或归档当前持久化结果；
- 不修改 PostgreSQL/Supabase 网络白名单或连接配置；
- 不为手机端建立独立页面、独立路由或独立数据状态；
- 不引入固定底部四项导航；
- 不在手机端隐藏样本、回答、评分、错误状态或持久化信息；
- 不把临时设计线框和 `.superpowers/` 工作目录提交到仓库。
