# 评测结论修复与兼容批次增量评测设计

## 1. 目标

本次改造修复当前结论聚合的两类误判，并在不修改评分模型、不重复调用现有三个模型的前提下，新增两个模型的完整评测和评分：

1. 评分说明只作为判断依据展示，不再通过文本关键词二次改变使用边界。
2. 单个高风险低分只把模型整体判断限制到“需谨慎参考”，不直接把整个模型判为“不可作为依据”。
3. 相同数据集、Prompt 和运行参数下的独立批次可组成同一比较组；同一模型和样本只采用最新成功评分。
4. 新增 `zai-org/GLM-5.2` 与 `Pro/moonshotai/Kimi-K2.6`，仍使用 `deepseek-ai/DeepSeek-V4-Pro` 作为固定裁判。
5. 新回答和新评分逐条写入 PostgreSQL/Supabase，任一持久化失败立即停止后续 Token 调用。

整个改造不修改现有项目静态文案、评分标准、裁判 Prompt、评分模型或 PostgreSQL 表结构。

## 2. 当前数据基线

当前 PostgreSQL/Supabase 中只有一套正式结果：

- 回答批次：`FRESH-20260717-CURRENT13-V1`；
- 评分批次：`FRESH-SCORE-20260717-CURRENT13-V1`；
- 13 个当前样本、3 个模型；
- 39 条成功回答、39 条成功评分；
- 无重复、无缺失、无孤儿记录；
- 数据集版本、数据集哈希和 Prompt 哈希与当前仓库完全一致。

这 39 条回答和评分保持原样，不更新、不复制到新批次，也不重新消耗 Token。

## 3. 已确认的根因

### 3.1 评分说明被重复解释

裁判已经通过五个维度和总分表达质量判断。当前结论服务又扫描 `review_note`，把“风险”“不足”等词当作负面信号。正面说明如“风险覆盖全面”也会触发降级，导致平均分 95.2 的 DeepSeek 被判为“需谨慎参考”。

修复后，`review_note` 和 `rationale` 继续进入模型详情和主要依据，但不再改变使用边界，也不再单独生成“评分说明提示需谨慎”。结构化分数、错误标签和任务风险仍参与判断。

### 3.2 单个低分直接触发最低档

当前全部 13 个样本均为高风险任务。现有规则只要发现一个低于 60 分的高风险样本，就把模型整体边界直接设置为“不可作为依据”。这会把局部失败扩大成全局结论。

修复后：

- 发现一个或多个高风险低分时，模型整体判断至少为“需谨慎参考”；
- 只有平均分低于 60、关键维度达到严重低分、或高风险任务存在结构化高严重度错误时，才判为“不可作为依据”；
- 具体低分样本继续保留在判断依据和模型详情中。

## 4. 兼容比较组

新增模型必须使用独立的 `run_id` 和 `score_run_id`，不能事后修改已经完成的旧批次。结论页通过兼容比较组组合新旧批次。

两个回答批次只有在以下元数据全部一致时才兼容：

- `dataset_version`；
- `dataset_hash`；
- `prompt_hash`；
- `generation_parameters_json`；
- `judge_parameters_json`。

结论服务以最新完成的正式运行作为当前比较组基准，选出所有兼容运行的成功 AI 评分。随后按 `(case_id, eval_model)` 去重，只保留 `updated_at`、`created_at` 和数据库 `id` 顺序上最新的成功记录。

这套规则具有以下效果：

- 当前三个模型继续使用原批次的 39 条结果；
- 新两个模型使用新批次的 26 条结果；
- 结论页形成 5 个模型、65 条评分的当前比较组；
- 后续重复评测某个模型时，历史记录保留，但当前结论只使用它最新的兼容成功评分；
- 数据集、Prompt 或参数变化后，不会把旧结果混入新口径。

如果运行元数据缺失或无法形成兼容组，结论页返回空结果或仅使用可验证的当前运行，不猜测兼容关系。

## 5. 新增模型选择

本次从 SiliconFlow 当前文本对话模型目录选择：

- `zai-org/GLM-5.2`；
- `Pro/moonshotai/Kimi-K2.6`。

选择理由是补充当前 DeepSeek、Qwen、LongCat 之外的两个独立模型家族，避免新增模型继续集中在同一技术系列。模型目录检查只验证可用性，不产生回答 Token。

评测参数保持与现有正式批次一致：

- 回答温度：`0.1`；
- 回答最大 Token：`4096`；
- 裁判模型：`deepseek-ai/DeepSeek-V4-Pro`；
- 裁判温度：`0.0`；
- 裁判最大 Token：`2048`；
- 样本：当前全部 13 个 active 样本。

## 6. 增量持久化流程

### 6.1 回答阶段

1. 从当前样本和 Prompt 构建新批次元数据并验证其与现有正式批次兼容。
2. 在任何模型调用前连接 PostgreSQL、执行 `ping` 并写入完整的 26 项回答队列。
3. 每项开始前标记为 `running`。
4. 每个回答完成后，在同一事务中写入 `live_run_responses` 并更新队列为 `success` 或 `failed`。
5. 数据库写入失败时立即中止，不调用下一项。
6. 中断恢复时只处理未成功项，不重新调用成功项。

### 6.2 评分阶段

1. 只为成功回答创建评分队列。
2. 在任何裁判调用前完整写入评分队列。
3. 每条评分完成后立即写入 `live_run_scores` 并更新评分队列。
4. 失败评分不进入结论；可通过现有恢复机制重试失败项。
5. 数据库写入失败时立即停止后续裁判调用。

API Key、数据库 URL、Authorization 头和完整供应商错误响应不写入仓库、数据库或终端输出。

## 7. 结论计算

静态页面文案和现有三档判断表述保持不变。修复后的预期旧模型判断为：

- DeepSeek-V4-Pro：`可作为初稿参考`；
- LongCat-2.0：`需谨慎参考`；
- Qwen3.6-35B：`需谨慎参考`。

GLM-5.2 和 Kimi-K2.6 的判断只能由真实评分动态产生，不预设结果。排序仍只是当前样本范围内按平均分的稳定展示顺序，不形成脱离样本范围的总体排名或采购建议。

## 8. 代码范围

主要修改：

- `app/services/conclusions.py`：兼容比较组读取、最新成功记录去重、边界规则修复；
- `src/ui/conclusions.py`：改用当前兼容比较组结果；
- `tests/test_conclusions.py`：批次组合、去重和空状态测试；
- `tests/test_model_use_boundaries.py`：正面“风险”文本不降级、孤立低分只限制到谨慎档、高严重度错误仍进入最低档。

不修改：

- `app/services/scorer.py` 中的裁判模型和评分 Prompt；
- 数据集题目、标准答案、评分维度和权重；
- PostgreSQL/Supabase 表结构；
- 现有页面静态文案；
- 当前 39 条回答和评分。

## 9. 测试与验收

按 TDD 实现，至少覆盖：

1. “风险覆盖全面”“无不可接受错误”等正面说明不会导致降级。
2. 中等平均分仍为“需谨慎参考”。
3. 高平均分加一个高风险低分只降到“需谨慎参考”。
4. 平均分低于 60、严重维度低分或高风险高严重度错误仍为“不可作为依据”。
5. 不兼容的数据集或 Prompt 批次不会合并。
6. 兼容批次中的不同模型会合并。
7. 同一模型和样本重复运行时只采用最新成功评分。
8. 失败、mock、skipped 和 inactive 记录不进入结论。
9. 全量 pytest、Ruff 和数据集校验通过。
10. PostgreSQL 契约测试通过，现有 39 条记录在代码部署前后保持不变。

## 10. 部署与正式运行顺序

1. 在独立分支完成测试驱动实现。
2. 运行全量本地与 PostgreSQL 验证。
3. 推送分支、创建 PR 并合并到 `main`，等待 GitHub CI 成功。
4. 确认远端 `main` 为合并提交。
5. 使用本地已配置的 SiliconFlow API Key 与 PostgreSQL URL 创建新两模型批次。
6. 完成 26 条回答后创建并执行评分批次。
7. 核对队列、回答、评分、样本覆盖、重复率和 Token 统计。
8. 从结论服务重新计算 5 模型判断，确认线上部署读取相同持久化数据。

模型调用阶段不依赖 Streamlit 会话；即使页面刷新或应用重启，已经成功写入 PostgreSQL 的结果仍可恢复。

## 11. 明确不做

- 不把新结果追加到旧 `run_id` 或旧 `score_run_id`。
- 不重新调用现有三个模型。
- 不复制现有回答或评分到新批次。
- 不修改评分模型、裁判 Prompt 或评分标准。
- 不删除、覆盖或归档现有正式结果。
- 不自动替换不可用的新模型；若目录或调用验证失败，保存失败状态并停止或按现有重试机制处理。
- 不创建新的历史档案页面或修改现有项目文案。
