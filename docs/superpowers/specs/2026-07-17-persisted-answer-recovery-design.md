# 持久化模型回答恢复与查看设计

## 1. 背景与根因

PostgreSQL/Supabase 中的模型回答没有丢失。当前共有两个正式回答批次：

- `FRESH-20260717-CURRENT13-V1`：39 条回答；
- `EXTEND-20260717-DIVERSE2-V1`：26 条回答。

65 条 `live_run_responses.answer_text` 均非空，最新批次也可以通过
`restore_compare_result_from_db(run_id)` 直接还原。

线上看不到回答由两个独立问题叠加造成：

1. “评测结论”页只读取当前兼容评分，没有读取或展示 `live_run_responses`。
2. “发起评测”页只自动恢复最新队列，并在展示前重新计算
   `dataset_hash` 与 `prompt_hash`。现有哈希生成依赖队列中重复的模型—样本组合，
   因此相同样本和 Prompt 在 3 个模型与 2 个模型的独立批次中会得到不同重算值。
   新批次为保持结论口径兼容，保存的是已验证的基线签名；页面重算后误判为不兼容，
   在读取已持久化回答之前提前返回空结果。

## 2. 目标

本次改造实现以下结果：

1. 页面重启或会话丢失后，可从 PostgreSQL/Supabase 查看已完成回答。
2. 用户可在两个正式批次之间切换，并查看每一条回答全文。
3. 当前兼容比较组中的 65 条回答都可通过模型与样本定位。
4. 已完成回答的只读展示不再受队列长度或模型数量导致的哈希重算影响。
5. 未完成批次的继续执行仍保留安全校验，不能因展示修复而误续跑不兼容任务。
6. 不触发任何模型或裁判调用，不改写现有回答、评分或运行元数据。

## 3. 不在本次范围内

- 不修改评分模型、裁判 Prompt、评分维度或评分边界。
- 不修改题库、专业标准答案或现有项目静态文案的表述。
- 不迁移、覆盖、复制或合并两个已有 `run_id`。
- 不修改 PostgreSQL 表结构。
- 不把 65 条回答伪装成同一个物理批次。
- 不增加后台任务或自动重跑。

新增的批次选择、查看按钮和空状态属于操作控件文案，不替换现有项目说明文案。

## 4. 方案比较

### 方案 A：移除现有哈希校验，仅恢复最新批次

优点是改动最小。缺点是只能看到最新 26 条回答，旧 39 条仍没有入口；同时若直接移除
所有校验，未完成队列可能在数据集变化后被错误续跑。

### 方案 B：区分“只读查看”与“继续执行”，增加批次选择器

已完成回答的查看直接使用数据库中的运行身份和响应，不重新生成兼容哈希；只有继续
未完成队列时才执行严格校验。页面列出所有有持久化回答的正式批次，默认选中最新批次，
用户可切换查看。评测结论页按评分记录的 `(run_id, case_id, eval_model)` 精确关联回答。

这是推荐并采用的方案。它保留批次不可变性，解决当前误拒绝，也不会放宽续跑安全边界。

### 方案 C：把全部兼容回答合成一个虚拟运行

可以一次显示 65 条，但会模糊物理批次身份，使重试、审计、Token 统计和运行元数据难以
解释。当前结论层已经能够组合兼容批次，回答层无需再创造第二套虚拟批次概念。

## 5. 架构设计

### 5.1 持久化回答查询服务

在 `app/services/eval_runner.py` 增加只读查询接口：

- 列出所有具有 `live_run_responses` 的运行；
- 为每个运行返回运行 ID、时间、状态、模型数、样本数、回答数与成功数；
- 按 `run_id` 恢复 `CompareRunResult`；
- 按运行时间和数据库顺序稳定排序，最新运行排在前面。

查询结果只来自 `live_evaluation_runs`、`live_run_queue` 与
`live_run_responses`，不依赖 Streamlit 会话。没有回答的空队列不进入历史查看列表。

### 5.2 查看与续跑分离

“只读查看已完成回答”和“继续未完成项”使用不同判定：

- 只读查看：运行存在、至少一条持久化响应、响应主键完整即可恢复。页面不重算
  `dataset_hash` 或 `prompt_hash`，因此不会被模型数量和队列长度误伤。
- 继续未完成项：保留当前元数据校验，并要求未完成队列能映射到当前样本。校验失败时
  只禁止继续执行，不影响查看已经保存的回答。

现有 `_recover_latest_run()` 改为先恢复最新有回答的运行，再为该运行建立页面状态。
严格 checkpoint 校验只控制“继续未完成项”按钮，不再控制回答是否可见。

### 5.3 “发起评测”页批次选择

“模型回答”区域新增持久化批次选择器：

- 默认选中最新有回答的批次；
- 选项显示运行时间、模型数、样本数和回答数；
- 切换批次后调用 `restore_compare_result_from_db(run_id)`；
- 继续复用现有“查看回答”选择器、回答预览与“查看全文”对话框；
- 当前会话刚生成的运行仍优先展示，不会被历史选择器覆盖。

批次切换只改变页面展示状态，不更新数据库，不触发评分或模型调用。

### 5.4 “评测结论”页回答入口

结论页继续使用 `load_current_cohort_scores()` 生成模型判断，同时读取
`live_run_responses` 作为只读明细。

在当前模型详情中提供回答选择与查看入口：

1. 先按当前选中模型过滤评分；
2. 以评分行的 `(run_id, case_id, eval_model)` 精确连接回答；
3. 选择样本后展示对应模型回答；
4. 长回答复用全文对话框或等价的只读 Markdown 详情组件；
5. 找不到回答时显示局部空状态，不影响评分结论。

不得只按 `case_id` 与模型名连接，因为同一模型可能在多个运行中重复出现。

## 6. 数据流

```text
PostgreSQL/Supabase
  ├─ live_evaluation_runs ─┐
  ├─ live_run_queue ───────┼─> 持久化批次摘要 ─> “发起评测”批次选择
  ├─ live_run_responses ───┘                         └─> 回答预览 / 全文
  └─ live_run_scores ─> 当前兼容评分 ─> 模型结论
                              └─(run_id, case_id, eval_model)─> 对应回答
```

两个物理批次保持不变；页面只是为它们提供只读索引和精确连接。

## 7. 错误处理

- PostgreSQL 不可用：沿用当前数据层不可用提示，不回退展示本地旧回答冒充线上结果。
- 运行存在但响应为空：不进入可查看批次列表。
- 某条回答为空或缺失：该条显示空状态，其余回答继续可见。
- 运行元数据不完整：允许查看已保存回答，但禁止继续未完成队列。
- 当前样本已删除或不可映射：回答仍可按保存的 `case_id` 查看；继续执行按钮禁用。
- 页面切换、刷新或 Streamlit 重启：重新从数据库构建批次列表和回答，不依赖 session。

所有读取异常都应失败为明确空状态，不打印数据库 URL、API Key 或完整供应商错误。

## 8. 测试设计

### 8.1 服务测试

1. 两个有回答的运行都进入批次列表，最新运行排在前面。
2. 空队列或无响应运行不进入列表。
3. 39 条与 26 条响应可分别按 `run_id` 完整恢复。
4. 已完成运行即使 checkpoint 重算不一致，仍可只读恢复。
5. 不兼容且未完成的运行不能继续执行。
6. 相同模型和样本跨运行时，回答连接必须使用 `run_id`，不得串批。

### 8.2 页面接线测试

1. “发起评测”页使用持久化批次列表和按 ID 恢复接口。
2. 现有回答选择、预览和全文查看组件继续使用。
3. “评测结论”页读取回答，并按三字段连接到当前评分。
4. 页面加载和批次切换不会调用模型 provider 或 scorer。

### 8.3 数据库与回归验证

1. PostgreSQL 只读核对仍为 65 条回答、65 条非空。
2. 两个运行分别为 39 条与 26 条，无重复主键。
3. 当前兼容评分仍为 65 条、13 个样本、5 个模型。
4. 全量 pytest、Ruff 和数据集校验通过。
5. 修复前后数据库行数和内容哈希不变。

## 9. 验收标准

- Streamlit 重启后，“发起评测”页可选择并查看两个正式批次。
- 39 条旧回答与 26 条新回答均能打开全文。
- “评测结论”页当前模型可定位到对应样本回答。
- 页面不再因为最新批次的重算哈希不同而隐藏已保存回答。
- 继续未完成队列仍受严格校验保护。
- 没有新增模型调用、Token 消耗、数据库写入或旧数据变更。
- 评分模型、评分结果、结论口径和现有项目文案保持不变。
