# 评测结果增量持久化与重新生成设计

## 1. 目标

本次改造同时解决两件事：

1. 将新的模型回答、裁判评分和执行队列逐条写入持久化数据库，确保页面刷新、应用休眠、进程重启或 Streamlit Community Cloud 重新部署后，不会因为重复调用模型而再次消耗 Token。
2. 持久化改造通过全部验收后，使用当前 13 个样本重新生成一批回答与评分，并确保每条高成本结果在继续下一项前已经落库。

整个改造不修改任何现有页面文案、项目介绍文案或业务字段表述。

## 2. 已确认的数据事实

- 当前 `app/db/findueval.db` 中运行回答和评分均为 0 条。
- 当前 `data/model_outputs.csv` 与 `data/scores.csv` 只有表头。
- Git 历史中的 45 条模型回答和对应评分基于旧版 15 个案例。它们与当前样本仅有 12 个相同案例 ID，但题目正文、业务背景、任务类型和标准答案均不完全一致，因此不导入正式结果。
- 更早的 15 条回答和评分是上述旧数据的完全相同子集，同样不导入。
- `data/demo_exports/demo_ai_scores.json` 中的 6 条记录虽然案例 ID 与当前样本一致，但没有对应回答且明确属于脱敏演示评分，因此保留原文件、不导入正式结果。
- 刚刚消失的 Streamlit 会话从未成功写入 SQLite，也没有导出文件、WAL 文件或其他副本，因此不能重建其回答正文。

## 3. 总体架构

保留现有静态数据和 SQLite 数据集仓储，不把全部数据层一次性迁移。新增一个只负责运行结果的 `ResultStore` 边界：

- 本地开发和测试默认使用 SQLite。
- 线上存在 PostgreSQL 连接配置时使用 PostgreSQL/Supabase。
- 两种后端共享同一组运行表、唯一约束、状态机和查询接口。
- 页面和模型服务不直接拼接 SQL，只调用结果存储接口。

建议使用 SQLAlchemy Core 实现小范围的双数据库兼容，并使用 PostgreSQL 驱动连接 Supabase。静态任务、标准答案和页面配置继续按现有方式从仓库数据与本地 SQLite 读取，避免扩大改造范围。

## 4. 数据模型

### 4.1 运行批次

新增 `live_evaluation_runs`，记录：

- `run_id`、运行状态和创建/更新时间；
- provider、模型列表、生成参数与裁判参数；
- prompt 版本及内容哈希；
- 数据集版本及内容哈希；
- 已完成、失败和待处理数量；
- 最后一次持久化错误。

该表不保存 API Key、Authorization 头或其他认证信息。

### 4.2 回答、评分和队列

继续使用现有四张运行表，并补充数据库级约束：

- `live_run_responses`：唯一键 `(run_id, case_id, model_name)`；
- `live_run_scores`：唯一键 `(score_run_id, case_id, eval_model)`；
- `live_run_queue`：唯一键 `(run_id, case_id, model_id)`；
- `live_score_queue`：唯一键 `(score_run_id, case_id, eval_model)`。

所有增量写入使用幂等 upsert。相同结果重复提交只更新同一行，不增加重复记录。

## 5. 运行与恢复流程

### 5.1 模型回答

1. 在任何模型 API 调用前创建运行批次，并一次性写入完整回答队列。
2. 只有队列持久化成功后才开始消耗 Token。
3. 每个任务调用开始时将对应项标记为 `running`。
4. 收到模型结果后，在一个事务内写入回答并将队列项更新为 `success` 或 `failed`。
5. 数据库写入失败时立即停止后续模型调用，不继续产生无法保存的结果。

### 5.2 裁判评分

1. 在裁判 API 调用前写入完整评分队列。
2. 每条评分完成后，在一个事务内写入评分并更新队列状态。
3. 已存在成功评分的组合不会再次调用裁判模型。

### 5.3 页面刷新与重启

页面重新进入时按 `run_id` 加载已持久化的回答、评分和队列：

- `success` 项直接复用；
- `running` 项在确认没有正在执行的进程后转为可重试状态；
- `queued` 与 `failed` 项可继续执行；
- 恢复执行只处理未成功项。

现有按钮、标题、说明文字和结果展示表述保持不变。

## 6. 基于当前样本重新生成

重新生成必须在 PostgreSQL/Supabase 持久化链路通过验收后执行：

1. 冻结本次运行使用的 13 个样本内容哈希、数据集版本、提示词哈希、模型列表和生成参数。
2. 在任何模型调用前创建正式运行批次，并将 `13 × 所选模型数` 条回答队列完整写入 PostgreSQL。
3. 按队列逐条生成回答；每条回答成功落库后才进入下一项。
4. 回答阶段完成后创建评分队列，使用当前固定裁判模型逐条评分并即时落库。
5. 中断后只继续未成功项，不重新请求已经成功的回答或评分。
6. 完成后核对回答数、评分数、成功/失败状态、样本哈希和 Token 统计，再允许导出和重启。

生成过程继续使用现有页面和文案，不增加新的数据分类。旧 45 条回答、45 条评分及 6 条演示评分都不写入新的正式结果数据库。

## 7. 配置与安全

连接优先级：

1. `DATABASE_URL` 环境变量；
2. Streamlit Secrets 中的数据库 URL；
3. 本地 SQLite 默认路径。

真实模型调用默认要求 PostgreSQL。只有显式设置 `FINDUEVAL_ALLOW_SQLITE_LIVE=1` 时，才允许本地使用 SQLite 执行真实模型调用；mock 流程和测试不受影响。这样线上即使遗漏 PostgreSQL 配置，应用仍可启动和浏览已有内容，但调用前的持久化预检会失败并停止，避免再次出现高成本结果只存在于会话内的情况。日志仅记录运行 ID、案例 ID、模型 ID、状态和异常类型，不记录密钥及完整提示词。

## 8. 错误处理

- 数据库不可用：阻止新的模型或裁判调用，保留已落库数据。
- 单条 API 调用失败：保存失败状态和可安全展示的错误信息，继续策略沿用现有流程。
- 单条数据库写入失败：停止当前队列，不把该项标记为完成。
- 重复提交：upsert 到原行。
- PostgreSQL 暂时断开：连接恢复后从持久化队列继续，不重新执行成功项。
- 当前样本内容与运行记录中的哈希不一致：阻止复用旧回答，并要求创建新运行批次。

## 9. 代码结构调整

新增聚焦模块：

- `app/persistence/result_store.py`：后端无关接口和领域异常；
- `app/persistence/sqlalchemy_store.py`：SQLite/PostgreSQL 实现；
- `app/persistence/schema.py`：运行结果表定义、唯一约束和索引；
- `app/services/run_checkpoint.py`：运行预检、样本/提示词哈希和断点状态协调。

现有 `eval_runner.py`、`scorer.py` 和 `src/ui/test_run.py` 只调整与持久化边界和恢复流程直接相关的代码。优先抽取现有静默吞错的持久化逻辑，不进行与本目标无关的大规模页面重写。

## 10. 测试与验收

按 TDD 实现，至少覆盖：

- API 调用前队列必须已成功持久化；
- 回答与评分逐条事务写入；
- SQLite 与 PostgreSQL 生成一致的业务行为；
- 重复写入不增加行数；
- 重启后只恢复未完成项；
- 持久化失败会阻止后续 Token 调用；
- 样本或提示词哈希变化后不会错误复用旧结果；
- 当前 13 个样本的新运行能够完整生成、评分并从数据库重建页面状态；
- 数据库配置不泄露到日志或页面；
- 所有原有 UI 文案守卫测试继续通过。

最终验收命令包括全量 pytest、数据集校验、Ruff、SQLite 集成测试、PostgreSQL 集成测试和 Streamlit 页面冒烟测试。完成验证后才停止旧实例并重启 Streamlit。

## 11. 明确不做

- 不尝试伪造或重新生成已经消失的会话回答。
- 不把旧 45 条回答、45 条评分或 6 条演示评分导入新的正式结果数据库。
- 不修改现有项目文案。
- 不创建单独的历史档案页面、分类或导出格式。
- 不把 API 密钥写入仓库或数据库。
- 不在本次改造中重写全部数据集 CRUD 或全部大型 UI 文件。
