# 数据集建设流程

本文说明如何把一个金融尽调任务沉淀为可测试、可评分、可复核的评测样本。当前项目支持两种方式：

- 在“样本库”页面通过样本管理区新增或编辑样本；SQLite 可用时会同步正式评测数据层。
- 按模板维护 `data/` 种子文件，适合版本化审阅和批量扩展。

两种方式都应遵守同一套质量门槛，见 `docs/dataset_quality_standard.md`。

## 1. 定义任务

样本应来自脱敏后的专业尽调场景，具备明确判定标准，能够区分回答质量。

需要维护：

- 任务编号；
- 场景；
- 任务题；
- 业务背景；
- 难度；
- 考察能力；
- 风险等级。

SQLite 中对应 `task_cases`；文件模式对应 `data/tasks.csv`。

## 2. 编写理想回复标准 / Gold Answer

Gold Answer 不是给被测模型看的答案，而是裁判评分与人工复核的评判锚点。

建议包含：

- 核心结论；
- 关键依据；
- 分析过程；
- 需核查材料；
- 边界条件；
- 必须覆盖点；
- 不可接受错误；
- 人工复核提示。

测试准入至少要求核心结论、必须覆盖点和不可接受错误齐备。资料不足时应写明需要进一步核验，不应伪造事实或依据。

SQLite 中对应 `gold_answers`；文件模式对应 `data/gold_answers.json`。

## 3. 维护 Rubric 评分标准

Rubric 用于把模型回答拆成可复核的评分维度。维度字段、名称和满分由统一配置维护，页面和裁判评分链路复用同一口径。

应维护：

- 评分维度；
- 满分；
- 满分标准；
- 扣分规则；
- 可关联的错误类型或说明。

SQLite 中对应 `rubrics`；文件模式主要来自 `data/dataset_manifest.yml` 和 `src/metrics.py` 的统一维度口径。样本管理中提供结构化 Rubric 时，会更新正式 Rubric 维度；普通文本只保留在管理视图中。

## 4. 标注错误标签与优化建议

错误标签用于解释模型可能出现什么问题，优化建议用于说明后续数据集如何补充或改进。

错误标签应记录：

- 错误类型；
- 严重程度；
- 错误表现；
- 修正方向。

优化建议应记录：

- 可能原因；
- 数据补强动作；
- 样本格式；
- 验证指标或预期效果。

对应对象为 `error_annotations` / `data/error_labels.csv` 和 `improvement_actions` / `data/optimization_plan.csv`。

## 5. 检查完整度与状态

样本不是手动设为“已入库”即可测试。可测试样本必须同时满足：

- 任务题、业务背景和场景存在；
- Gold Answer 具备核心结论、必须覆盖点和不可接受错误；
- Rubric 评分标准存在；
- 状态为已入库；
- 未归档或停用。

不完整样本可保存为待复核或需优化，但不会出现在“发起测试”页的可测样本列表。

## 6. 运行模型回答

发起测试时，被测模型只看到任务题、业务背景和输出要求。Gold Answer、必须覆盖点、不可接受错误和 Rubric 不会发送给被测模型。

未配置真实模型服务时，应用进入 mock 回退模式。mock 回答只用于打通流程，不代表真实模型结果。

## 7. 生成评分草稿

裁判模型读取任务、模型回答、Gold Answer 和 Rubric，生成建议分。评分写入后默认为待人工复核，不自动进入正式结论。

## 8. 人工复核与正式结论

复核页用于对照模型回答、Gold Answer、评分矩阵、错误归因和红线提示。人工确认或修订后，评分才会归档进入正式结论。

正式结论只代表当前样本内观察，不是模型排行榜。

## 扩展方式

新增样本可从 `templates/new_case_template.yml` 开始，也可以直接通过“样本库”页面维护。批量扩展后应运行：

```bash
python scripts/validate_dataset.py
```

校验通过后，再进入“样本库”检查完整度，并在“发起测试”页做小范围回归。
