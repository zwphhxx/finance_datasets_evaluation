# 财务/法律/投行场景大模型对比评测

财务/法律/投行场景大模型对比评测是一个面向财务、法律、投行等专业场景的模型评测样本库原型，用于通过专业任务样本、模型对比、专业标准答案、评分标准和人工确认，评估模型回答质量、主要问题和使用边界。

项目不包含真实公司数据，不引入登录、权限系统或外部数据库。当前定位是面试演示原型：流程完整、边界清楚、可解释，但不是完整平台系统。

## 当前主流程

当前用户主链路只有四步。项目说明页是一页专业 brief，只解释项目定位、评测流程和数据边界，不承担页面跳转：

1. **样本库**：维护任务题、业务背景、专业标准答案、中文业务状态和必要复核备注；完整且已入库的样本才可进入发起评测。历史模型回答、错误标签和数据优化建议为只读历史信息，不在新增 / 编辑样本时填写。
2. **发起评测**：选择可测样本和硅基流动模型，生成模型回答；被测模型只看到任务题、业务背景和输出要求。
3. **评分确认**：裁判模型逐条生成建议分、维度评分依据和复核提示，写入 SQLite 后由人工确认生效、修订后确认或暂不采用。
4. **评测结论**：只汇总已确认生效的真实运行评分，形成当前样本内观察。

评测边界固定如下：

- 被测模型不看到专业标准答案、必须覆盖点、不可接受错误或评分标准。
- 裁判评分只是评分草稿；确认生效后才进入正式结论。
- 待确认草稿、暂不采用记录和示例评价不进入正式结论；seed 示例评价不等于正式结论。
- SQLite 是运行期数据；Streamlit 重新部署可能丢失 live 结论，如需保留演示结论请使用导出 / 导入。

## 数据对象与数据流

主要对象如下：

| 对象 | SQLite 表 / 文件 | 说明 |
| --- | --- | --- |
| 任务题与业务背景 | `task_cases` / `data/tasks.csv` | 任务编号、专业场景、题干、背景、难度、风险等级和状态。 |
| 专业标准答案 | `gold_answers` / `data/gold_answers.json` | 标准结论、关键依据、必须覆盖点、不可接受错误、边界与需核查事项和人工复核提示。 |
| 评分标准 | `rubrics` / `data/dataset_manifest.yml` | 评分维度、满分、满分标准和扣分规则。 |
| 模型回答 | `model_responses`、`live_run_responses` / `data/model_outputs.csv` | 当前 seed 仅保留表头；真实运行回答写入 live 表。 |
| 评分记录 | `score_records`、`live_run_scores` / `data/scores.csv` | 当前 seed 仅保留表头；评分草稿和已确认评分写入 live 表。 |
| 可恢复队列 | `live_run_queue`、`live_score_queue` | 记录模型回答和评分草稿的 queued/running/success/failed 状态，便于中断后继续或重试。 |
| 错误归因 | `error_annotations` / `data/error_labels.csv` | 错误类型、严重程度、错误表现和修正方向。 |
| 数据优化建议 | `improvement_actions` / `data/optimization_plan.csv` | 从错误标签收敛出的数据补强动作。 |
| 样本视图 | `data/samples.json` | 中文业务状态、备注、导入导出和兼容管理信息。 |

数据源层级固定为：

- **seed 文件**：`data/tasks.csv`、`data/gold_answers.json`、manifest 评分标准等文件只用于初始化或数据库不可用时的兼容回退。
- **samples.json**：样本库管理视图、导入导出备份和中文业务状态记录，不作为发起评测的唯一正式源。
- **SQLite**：`task_cases`、`gold_answers`、`rubrics` 是正式评测资产；`live_run_responses`、`live_run_scores` 是真实运行结果和评分确认结果；`live_run_queue`、`live_score_queue` 保存可恢复运行状态。

SQLite 初始化后，发起评测和正式结论优先从 SQLite 读取。未初始化数据库时，项目仍可从 `data/` 种子文件浏览演示内容，但样本库新增或导入的样本不一定能进入发起评测。部署环境中的本地 SQLite 适合单机原型和演示，不应被描述为多人协作的长期持久化能力。

## 模型结果来源

当前 `data/model_outputs.csv`、`data/scores.csv`、`data/error_labels.csv`、`data/optimization_plan.csv`、`data/evaluation_runs.csv` 等运行结果 seed 文件仅保留表头，不再提交旧示例回答、旧评分或旧错误归因。`dataset_manifest.yml` 中保留的模型范围只用于数据校验元数据，不代表当前存在 seed 模型结果。

## 样本库与正式评测资产

SQLite 可用时，样本库新增、编辑、移出测试和 CSV 导入会同步维护正式评测数据层：

- `task_cases`：写入或更新样本编号、专业场景、难度、任务题、业务背景和底层状态。
- `gold_answers`：写入或更新专业标准答案，并保持 `raw_json` 与结构化字段一致。
- `rubrics`：正式评分标准由数据层统一维护，包含评分维度、满分、满分标准和扣分规则；新增 / 编辑样本时不要求普通用户维护评分标准技术字段。
- `samples.json`：继续作为轻量管理视图、导入导出和兼容备份；正式测试准入和评测内容以正式数据层为准。

新增、编辑、移出测试和 CSV 导入完成后，会校验正式资产同步状态：`task_cases` 中存在该 `case_id`，`gold_answers` 中存在对应记录，`rubrics` 中存在可用评分标准；如果样本为“已入库”，还需通过正式准入检查。同步失败时页面不会显示保存成功，会保留错误原因供修正。

样本库正常同步时不展示技术化数据源提示；仅在 SQLite 不可用、已入库样本未完成正式资产同步或同步失败时显示必要提醒，避免演示页面暴露运行期实现细节。

样本库页面提供弱入口“同步样本库”，用于将 `samples.json` 管理视图批量同步到 SQLite 正式资产。该操作只写入任务题、专业标准答案和评分标准引用，不覆盖 `live_run_scores`，不会删除历史评分。

样本库页面顶部提供 CSV 导入入口。默认 CSV 模板为简版字段：`case_id`、`title`、`professional_scene`、`status`、`question`、`context`、`output_requirement`、`standard_conclusion`、`key_evidence`、`must_have_points`、`unacceptable_errors`、`boundary_and_check_items`、`difficulty`、`risk_level`、`manual_review_notes`、`reviewer_note`、`scoring_focus`。其中 `professional_scene` 只允许“财务场景 / 法律场景 / 投行场景”。上传后会先校验必要字段；字段缺失、文件内编号重复或样本编号重复时会给出提示。重复样本可选择跳过、更新已有样本或取消导入。SQLite 可用时，导入后会同步正式资产；SQLite 不可用时，页面会明确提示“已写入样本库视图，但未写入正式评测资产”。旧版完整模板仍兼容导入，但不作为默认模板。

页面只展示中文业务状态，底层状态用于测试准入：

| 页面状态 | 底层状态 | 测试准入 |
| --- | --- | --- |
| 待复核 | `draft` | 不可测试 |
| 已入库 | `active` | 仍需通过完整度校验 |
| 需优化 | `draft` | 不可测试 |
| 已移出测试 | `inactive` | 不可测试 |

可测试样本必须同时满足：正式题库存在任务题、业务背景和专业场景；专业标准答案具备标准结论、必须覆盖点和不可接受错误；评分标准具备评分维度、满分、满分标准和扣分规则；状态为已入库且未移出测试。

评分标准只有维度和满分时，只能视为“评分维度配置”，不视为完整评分标准。样本库、发起评测和评分确认页会按同一口径展示缺失项；缺少满分标准或扣分规则的样本不会进入发起评测。

如果样本库中某条样本显示为“已入库”，但发起评测页不可见，优先检查样本详情中的完整度和同步状态。常见原因是专业标准答案或评分标准未同步到正式资产，或运行环境重建后 SQLite 回到 seed 初始状态。

删除在当前版本中不作为主流程能力开放。高风险样本处理优先使用“移出测试”：移出后样本不会进入发起评测，历史评测记录仍保留，避免破坏确认和结论追溯。

## 模型与裁判输入隔离

这是项目可信度边界：

- 被测模型只看到任务题、业务背景和输出要求。
- 被测模型不看到专业标准答案、必须覆盖点、不可接受错误或评分标准。
- 裁判模型才读取专业标准答案、评分标准和模型回答。
- 裁判评分只是评分草稿，不是最终结论。
- 人工确认或修订后，评分才会纳入正式结论；确认、修订或暂不采用后，页面会保留处理反馈并切换到下一条待处理评分。
- 评分确认页用评分批次区分不同时间生成的草稿；批次摘要展示待处理数量、已确认数量、覆盖模型和覆盖样本，评分详情仍由表格行选择。

## 正式结论口径

本项目不做模型排名。结论只代表当前样本内观察，不代表模型整体能力或采购建议。

正式结论只包括 `live_run_scores` 中已确认生效的真实运行评分。当前 `data/scores.csv` 与 `score_records` 仅作为空 seed 结构存在，不进入当前主流程结论。

待确认草稿、暂不采用记录和示例评价不进入正式结论。模型当前判断不只看平均分，还结合红线错误、关键维度短板、高风险任务表现、样本数量和人工复核说明。结论页只保留摘要，详细评分依据在评分确认页查看。判断表达统一为：

- 可作为初稿参考；
- 必须人工复核后使用；
- 暂不建议作为判断依据；
- 样本不足，暂不形成判断。

## 历史评分持久化与演示恢复

评测结论页读取的是运行期 SQLite 中的真实评分结果。`app/db/*.db` 和 `*.sqlite3` 保持在 `.gitignore` 中，不随 Git 提交。SQLite 是运行期状态，适合真实运行、现场确认和本地演示，不适合作为版本化数据资产。

重新部署、重启或重建 SQLite 后，`live_run_scores` 中的评分草稿和已确认结论可能消失。此时页面显示“暂无已确认评分”，通常表示当前运行环境没有可读取的 confirmed live 评分，不代表 seed 示例评价被删除。

如需保留或复现演示结论，请在“评测结论”页打开“数据维护”，导出已确认评分。导出文件为 UTF-8 JSON，`export_type` 为 `confirmed_score_export`，默认只包含 `review_status = confirmed` 且 `judge_status = success` 的真实运行评分；可选择同时包含待确认草稿。导出内容不包含 API Key、Authorization、请求头、模型服务密钥或异常堆栈。

可将人工检查后的脱敏 JSON 放入 `data/confirmed_score_exports/` 作为演示恢复数据。仓库提供 `data/confirmed_score_exports/demo_confirmed_scores.json` 作为格式基准；如需展示稳定历史结论，可用真实确认结果导出的脱敏文件替换其中 records。不要提交 `app/db/findueval.db` 或其他 SQLite 数据库。

在新的运行环境中，可在“评测结论”页打开“数据维护”，上传项目导出的评分 JSON，或点击“从演示评分文件恢复”。导入会写入 `live_run_scores`，重复记录按 `score_run_id + case_id + eval_model` 判断，可选择跳过或更新。导入不会删除现有评分。导入后，只有已确认评分会纳入正式结论；待确认、暂不采用、评分失败和 seed 示例评价仍不会进入正式结论。

## 运行中断与恢复

发起评测页不是后台任务系统。模型回答和裁判评分都在当前 Streamlit 页面执行；运行过程中不建议刷新、关闭页面或切换页面。页面刷新或浏览器关闭可能中断尚未开始或正在等待返回的外部 API 调用。

为降低中断影响，发起评测会在开始前把模型回答队列写入 SQLite `live_run_queue`，评分前把评分队列写入 `live_score_queue`。每条模型回答完成后会立即写入 `live_run_responses`，每条评分完成后会立即写入 `live_run_scores`。如果页面中断，已经完成的回答和评分会保留；未完成项保留在队列中，返回发起评测页后可以继续未完成项或重试失败项。

既有运行期 SQLite 在首次使用队列能力时会补齐缺失的队列表；该步骤只创建缺失表，不重建数据库，也不删除已有样本、回答、评分或确认结果。

“继续未完成项”只处理 `queued` / `running` 且尚未成功的队列项，不重新执行已成功项。“重试失败项”只处理 `failed` 队列项；模型回答重试不会重跑已成功回答，评分重试不会重新生成模型回答。放弃本次运行或评分只清理当前页面状态，不删除已成功写入 SQLite 的回答、评分或历史确认结果。

运行期 SQLite 仍不随 Git 提交。重新部署后如需稳定展示历史结论，应使用“已确认评分导出文件 + 数据维护导入/恢复”的方式恢复。

## 建议演示路径

1. 打开“项目说明”，看项目定位、评测闭环和入口。
2. 进入“样本库”，查询样本索引，在页面顶部新增样本或导入 CSV。
3. 在“当前样本”详情窗格查看专业标准答案、评分标准和完整度，编辑当前样本或将样本移出测试。
4. 进入“发起评测”，在评测配置中选择已入库且完整的样本。
5. 通过模型弹窗选择硅基流动可用模型；运行参数采用固定评测配置。
6. 运行模型回答，查看运行结果摘要和明细。
7. 生成评分草稿，查看逐条评分、维度依据和复核提示。
8. 进入“评分确认”，在待处理列表中选择一条评分，查看摘要和评分依据后确认生效、修订后确认或暂不采用。
9. 人工确认或修订后确认生效，暂不采用的评分不会进入正式结论。
10. 到“评测结论”查看当前结论、模型当前判断、当前模型详情和待确认评分提醒。

演示不依赖固定样本、固定模型或固定分数。

## 本地运行

建议使用 Python 3.11 或以上版本。

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
streamlit run app.py
```

如需使用自定义种子数据目录：

```bash
export FINDUEVAL_DATA_DIR=/path/to/data
streamlit run app.py
```

## SQLite 初始化

项目不初始化数据库也能运行；此时读取 `data/` 种子文件。启动时会默认检查本地 SQLite：如果数据库不存在或不可用，会从种子数据自动初始化一次；如果数据库已可用，不会重复覆盖，避免清空运行时样本、评分草稿和人工复核记录。

如需手动初始化或重建，可执行：

```bash
python -m app.db.init_db
python -m app.db.init_db --force
python -m app.db.init_db --db /path/to/findueval.db --data-dir /path/to/data
```

自定义数据库路径：

```bash
export FINDUEVAL_DB_PATH=/path/to/findueval.db
streamlit run app.py
```

如需关闭部署启动时的自动初始化：

```bash
export FINDUEVAL_AUTO_INIT_DB=0
streamlit run app.py
```

SQLite 数据层仅使用标准库 `sqlite3`。相关文件：

- `app/db/schema.sql`：任务题、专业标准答案、评分标准、模型回答、评分、错误标签、优化动作、评测运行和 live 结果表。
- `app/db/init_db.py`：从 `data/` 初始化数据库。
- `app/db/repository.py`：基础读写封装。
- `app/services/dataset_service.py`：SQLite 优先、文件回退的数据服务层。
- `app/services/sample_repository.py`：样本视图与正式数据层同步。

当前 SQLite 仅适合单机演示和面试原型。Streamlit Cloud 等部署环境的本地文件系统不应被视为长期多人协作数据库；需要多人标注、权限或审计时，应另行设计协作后端。

## 模型服务与安全

按 `st.secrets`、环境变量、`.env` 的顺序读取配置。可参考 `.env.example`：

| 配置项 | 说明 |
| --- | --- |
| `SILICONFLOW_API_KEY` | 硅基流动模型服务 API Key；缺失时发起评测页不能发起真实调用。 |
| `SILICONFLOW_BASE_URL` | API Base URL。 |
| `SILICONFLOW_TIMEOUT_SECONDS` | 请求超时秒数。 |
| `FINDUEVAL_AUTO_INIT_DB` | 是否在启动时自动初始化 SQLite；默认启用，设为 `0` 可关闭。 |

安全边界：

- 不要提交 API Key。
- API Key、Authorization 请求头和完整请求头不应展示在页面、日志或错误信息中。
- 后端保留模拟回退用于开发和自动化验证，但不在发起评测页作为可选模型服务展示。
- 模拟回退只用于打通链路，不冒充真实模型结果。

## 运行稳定性与失败恢复

模型回答和评分依赖第三方模型服务，实时调用不能保证 100% 成功。网络波动、模型服务响应慢、限流、被测模型回答较长或裁判 prompt 较长，都可能导致 timeout；单条失败不代表样本失败，也不代表整批运行失败。
评分失败通常是裁判模型超时，不代表样本失败，也不代表被测模型回答失败。

`SILICONFLOW_TIMEOUT_SECONDS` 控制单次请求等待时间。普通模型建议 60-90 秒；LongCat、R1、reasoning / thinking 类慢模型或全量运行建议 120-180 秒。timeout 过长会导致页面等待时间变长，并不能保证外部服务一定成功。

可在本地环境或部署 Secrets 中按演示需要调整，例如：

```bash
SILICONFLOW_TIMEOUT_SECONDS=120
```

Streamlit Secrets 示例：

```toml
SILICONFLOW_TIMEOUT_SECONDS = "120"
```

该配置仍可按环境调整；非法配置值会回退默认超时。页面错误信息会继续提示可调整 `SILICONFLOW_TIMEOUT_SECONDS`。

失败后的处理方式：

- 模型回答 timeout 会在服务层自动重试 1 次；`finish_reason=length` 会用更高输出 token 预算自动重试 1 次。
- 可重试的裁判评分错误会在服务层自动重试，最多重试 2 次。
- 单条模型回答失败不会中断整批运行，已成功回答会逐条写入 SQLite。
- 单条评分失败不会中断整批评分，已成功评分会逐条写入 SQLite。
- 发起评测页提供“重试失败项”，只重试失败的 `(样本, 模型)` 回答，不重新生成已成功回答。
- 发起评测页提供“重试失败评分”，只重试裁判失败的评分项，不重新生成模型回答，也不覆盖已确认评分。
- 已确认评分才进入正式结论，失败评分不会进入评分确认待处理队列，也不会进入评测结论。

发起评测页默认支持批处理运行，用户选择 13 个样本和多个模型时可以直接运行。运行队列会展示进度、等待项和已完成结果；单条失败不会删除已完成回答，可继续未完成项或重试失败项。为了保证现场演示稳定，建议提前生成并确认评分结果，再在评测结论页打开“数据维护”导出已确认评分；演示前可通过上传评分 JSON 或“从演示评分文件恢复”恢复已确认结果。现场实时评分失败时，仍可展示已确认历史结论。

## Streamlit Cloud 部署

1. 将仓库推送到 GitHub。
2. 在 Streamlit Community Cloud 选择该仓库。
3. 设置主文件为 `app.py`。
4. 确认依赖来自 `requirements.txt`。
5. 如需真实模型调用，通过 Streamlit Secrets 配置密钥。
6. 不配置数据库或模型密钥也能启动；未配置密钥时可以浏览页面和样本，但不能发起真实模型调用。
